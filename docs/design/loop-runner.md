# neo Loop Runner - Design Spec

A spec for a bounded, verified, reportable loop feature in neo. It also resolves the
goal-vs-loop confusion, because the resolution is what determines the design.

---

## 0. Goal vs loop, resolved

The confusion is real and it has two causes.

**Cause one: "loop" is two different words.** There is the general idea (an agent loop: iterate, check, decide, repeat until done) and there is Claude Code's specific `/loop` command (a timer that re-runs a prompt every N minutes). They are not the same thing. A `/goal` is also "an agent loop" in the general sense.

**Cause two: the tools differ, and not symmetrically.**

| | Claude Code | Codex |
|---|---|---|
| `goal` (run until a condition is verified) | yes, `/goal` | yes, `/goal` (behind a flag) |
| `loop` (re-run a prompt on a timer) | yes, `/loop` | **no** |
| schedule (run while away) | scheduled agents | Automations |

So both have a goal. Only Claude Code has a timer-loop. Codex has none. The impression that "both have both" is the confusion itself.

**The unifying model.** Goal and loop are not two engines. They are one iterate-check-decide engine that differs on exactly two axes:

- **Trigger:** what starts the next iteration. A goal's trigger is "the previous turn finished and the condition is not yet met." A timer-loop's trigger is "a clock fired." A routine's trigger is "a schedule fired while you were away."
- **Stop condition:** what ends the whole run. A goal stops when a verifier passes. A timer-loop stops when you stop it, or a budget caps it; it does not check whether it succeeded. A routine stops per fire on a budget.

This trigger-and-stop framing is the same one Claude Code's own docs use to compare `/goal`, `/loop`, and Stop hooks: they tabulate each by "next turn starts when" and "stops when." neo just takes that to its conclusion: if those are the only two axes, you need one engine, not three commands.

Said plainly: a **goal is a loop whose stop condition is a verifier.** A **timer-loop is a loop whose trigger is a clock.** A **routine is a loop whose trigger is a schedule.** Same engine, different trigger and stop.

**neo's decision.** Build one runner, not three commands. A loop is a declared, bounded job (a file). Whether it behaves like a goal, a timer-loop, or a routine is configuration: the trigger and the stop condition. This is the opinionated choice that removes the confusion, instead of importing it from the other tools.

---

## 1. Design principles

1. **A loop is a definition, not a command string.** `neo goal "fix the tests"` puts the job in whatever you typed. `neo loop run docs-drift` points at a file that already carries the prompt, the verifier, and the bounds. The bounds live with the job.
2. **Never run forever.** The runner refuses to start without at least one budget limit.
3. **The verifier is a shell command, a prompt, or both.** Not everything reduces to exit code 0. "The tests pass" is a command; "the guide reads clearly and invents nothing" is a prompt. Both are first-class.
4. **Skip when there is nothing to do.** Run the verifier once before iterating; if it already passes, exit without burning a turn.
5. **Emit a machine-readable report.** A loop must be drivable by CI or another loop, so the result is JSON with a status, not just printed text.
6. **State on disk.** Each iteration persists a snapshot, so a run can be resumed after a crash.
7. **Loops are reusable artifacts, in two scopes.** A loop lives in the repo (`.neo/loops/`) or in your home directory (`~/.neo/loops/`), the same way neo already layers config and skills. The file is the durable, shareable, version-controlled object; running it is just `neo loop run <name>`.
8. **Described in English, generated by the agent, bounded by a human.** You write a plain sentence; the agent drafts the loop file; you confirm the two lines that matter (the verifier and the budget) before it saves. The agent translates, it does not get to decide the bounds.
9. **Two front doors, one runner.** `neo loop run <name>` (CLI, prints JSON) and `/loop <name>` (in-session, prints a live status and a human report) drive the same engine.

## 2. The loop definition file

One YAML file per loop, in `.neo/loops/<name>.yaml` (project-local) or `~/.neo/loops/<name>.yaml` (global). This is the unit of work. See section 3 for how the two scopes resolve.

```yaml
name: docs-drift

# Required. One line, max 100 chars. This is the label shown in `/loops` and
# `neo loop list`, and the human-readable contract for what the loop is for.
# It is the one field the tool cannot synthesize, so it is mandatory.
description: "Fix one doc that no longer matches the code"

# What the worker does each iteration. The full plain-English brief.
# A strong objective names six things (OpenAI's Codex Goals cookbook calls these
# the goal contract): the outcome, the verification surface, the constraints, the
# boundaries, the iteration policy, and the blocked stop condition. The first
# three and the last are covered by `verify`, `boundaries`, and `on_blocked`
# below; state the outcome and the iteration policy here.
prompt: |
  Outcome: every doc under /docs matches the code, one fix at a time.
  Find one doc that no longer matches. Fix only that doc; do not change code.
  Iteration policy: after each fix, note what was wrong and what you changed,
  then pick the next most-drifted doc. Show the diff.

# What may be touched. A boundary the run enforces, not a suggestion. Paths the
# worker may write to; everything else is read-only for this loop. (optional)
boundaries:
  write: ["docs/**"]
  forbid: ["**/*.go", "go.mod"]

# What triggers the run.
run:
  trigger: manual          # manual | interval | schedule
  interval: 5m             # used when trigger=interval
  schedule: "0 8 * * 1-5"  # cron, used when trigger=schedule

# The stop condition. At least one of command/prompt is required.
verify:
  command: "go test ./... && ./scripts/check-docs.sh"  # exit 0 = pass (optional)
  prompt: |                                            # model judgment (optional)
    Does the fixed doc match the code with no invented behaviour, and read clearly?
    Answer PASS or FAIL on the first line, with one concrete reason.
  model: claude-haiku-4-5   # the verifier model; cheaper than the worker

# Bounds. Required. The run refuses to start if all are empty.
budget:
  max_iterations: 3
  max_cost_usd: 5
  max_minutes: 20

# Safety and preconditions.
require_clean_worktree: true
worktree: false            # true = run inside `git worktree add` for isolation
permissions: trusted       # trusted | ask | readonly (use readonly when unattended)

# Stop behaviours.
skip_if_passing: true      # run verify once first; exit nothing-to-do if already green
on_blocked: stop           # stop | escalate ; triggers after the same failure 3 turns
```

Run state is not stored here; it lives next to the definition, scoped to match it (section 3). CLI flags override file fields for a one-off run (`--max-iterations`, `--max-cost`, `--require-clean-worktree`).

**Validation.** A file fails to load, and cannot run, unless it has: a `name`, a one-line `description`, a `prompt`, at least one `verify` gate (`command` or `prompt`), at least one `budget` limit, and a valid `trigger`. This is what stops "the agent wrote it" or "someone hand-edited it" from ever meaning "the stop condition got skipped."

**The six-part objective.** A goal is more than a bigger prompt; the strongest objectives name six things, and the schema maps to all of them: outcome and iteration policy (in `prompt`), verification surface (`verify`), constraints and boundaries (`boundaries`), and the blocked stop condition (`on_blocked` plus the budget). Two of these are worth calling out because they are the levers a freeform prompt usually misses. **Boundaries** are enforced, not advised: `write` is the set of paths the worker may modify this run, and the runner denies writes outside it at the tool layer, so "do not change code" is a permission, not a polite request. **Iteration policy** is the line that prevents thrashing: it tells the worker how to choose the next action after each attempt (record what changed, then pick the next experiment), which is the difference between progress and re-trying the same broken approach.

## 3. Scopes: local and global

A loop lives in one of two places, mirroring how neo already layers config and skills:

- **`.neo/loops/<name>.yaml`** is project-local. It is committed with the repo and shared with the team. A loop with a repo-bound verifier (`go test ./...`) belongs here.
- **`~/.neo/loops/<name>.yaml`** is global and personal, available in every repo. A loop with no repo-bound verifier (a changelog watcher, a morning triage) belongs here.

**Resolution.** `neo loop run docs-drift` searches local first, then global. Local wins on a name clash, because the repo-specific version is the more specific intent. To force the global one, `neo loop run docs-drift --global`.

**Run state follows the loop's scope.** A local loop's state goes under `.neo/loops/state/<name>.json`; a global loop's under `~/.neo/loops/state/<name>.json`. A personal loop run inside a repo does not litter the project tree.

**Trust, because a local loop is executable code that arrives with a clone.** A loop file carries a prompt, a budget, and a shell command in its verifier. Cloning a repo and running its loop runs that shell command. So treat a project-local loop like any untrusted code:

- On the **first run** of a project-local loop (one neo has not seen before), show its `verify.command` and its `permissions`, and require confirmation before running.
- The file's `permissions` is a ceiling the run cannot exceed. A `readonly` loop cannot be made to write because the prompt asks it to.
- **Global loops are yours**, so they do not need the first-run gate.

This matters more precisely because the local scope exists to let people share loops, which is the point.

## 4. Authoring: `neo loop new`

Nobody should hand-write the YAML. You describe the loop in English; the agent drafts the file; you approve the gate and the budget.

```
neo loop new "fix any doc under /docs that no longer matches the code, one at a time"
```

The agent inspects the repo (is there a test command, a smoke script, a git remote), then drafts a definition and shows it back. What it may and may not decide is the whole design:

**The agent drafts:** the `name`, the one-line `description`, a `prompt` structured into the six-part objective (outcome, verification surface, constraints, boundaries, iteration policy, blocked stop condition), proposed `boundaries`, and a *proposed* verifier. It should prefer a command verifier and go looking for one, because "fix the docs" has no exit code but `go test` does. Finding the strongest available gate is its main job, not defaulting to a prompt.

**The human confirms:** the verifier and the budget. These are the two lines that decide whether the loop is safe, so they are never silently chosen. The agent proposes conservative budget defaults; you raise them.

**The agent must be honest about the gate.** When only a prompt verifier is possible, it says so and marks the loop as needing human review, not autonomous. That honesty is the feature, not a limitation.

**Default scope is local**, since most generated loops are about the repo you are in, and a committed loop is more useful than a hidden personal one. `--global` writes to `~/.neo/loops/`. The agent can *suggest* global when the verifier is not repo-bound, but it suggests; it does not silently choose.

The interview is mostly inference, confirmed in one screen:

```
Drafting loop "docs-drift"  ->  .neo/loops/docs-drift.yaml (local)

  description: Fix one doc that no longer matches the code
  prompt:      (generated from your sentence)
  verify:      command "go test ./... && ./scripts/check-docs.sh"   <- found these
               + prompt "doc matches code, no invented behaviour"    <- judgment part
  budget:      max_iterations 3, max_cost_usd 5                      <- defaults, raise if needed
  trigger:     manual

  Note: part of this is verified by a model's judgment, so this is a
  draft loop. Review its output before it merges.

Save? [y / edit / n]   (use --global to save to ~/.neo/loops)
```

The original sentence is kept as the `description` (or expanded into `prompt`), so the file stays self-documenting: it is what someone reads first, and what regenerates or edits the loop later. The same validation from section 2 runs on save, so a generated file that lacks a verifier or a budget cannot be written.

This describe-then-tighten flow is not a neo invention. OpenAI's Codex Goals cookbook recommends the same two steps: describe the work in plain language and have the model turn it into a draft goal, then review and tighten the success condition, verification surface, constraints, and blocked stop before activating it. neo's only addition is making the budget and the command verifier things the human must confirm, not things the model gets to set.

## 5. The verifier (two kinds, run command-first)

A loop is only as good as its verifier, so this is the most important part.

- **Command verifier.** Objective and cheap. Exit 0 is pass. It cannot be argued with or hallucinated. Use it whenever the condition reduces to a command.
- **Prompt verifier.** A judgment, for conditions that do not reduce to a command. It carries a failure mode the command does not: a model can be lenient, inconsistent, or gamed. Two rules keep it honest, the same ones Claude Code's separate evaluator and Codex's audit prompt rely on. It must be a **separate model call** from the worker, so the agent is not grading its own work. And it must judge **against evidence in the transcript**, forced to PASS or FAIL with a concrete reason, told to treat completion as unproven. A prompt verifier that says "looks good" is a second optimist, not a gate.

**Run order.** If a command verifier is set, run it first. If it fails, fail immediately and do not spend the model call. Only if the command passes (or is absent) and a prompt verifier is set, run the prompt. If both are absent, the definition is invalid.

**Trust is not uniform.** "Passed by the test suite" and "passed by a model's judgment" are different levels of confidence, so the report records which gate decided it (`verified_by`). A loop whose only gate is a prompt verifier is a draft loop with a human review step, not a fully autonomous one. State that in the course and in the report.

## 6. The run algorithm

```
load definition + prior state (if resuming)
if budget has no limits:                refuse, exit
if require_clean_worktree and dirty:    refuse, exit
if worktree: run inside `git worktree add` for this loop
if skip_if_passing and verify() passes: report(nothing_to_do), exit

directive = definition.prompt
for i in 1..max_iterations:
    if budget exceeded (iterations/cost/time):  report(budget_limited), exit
    if ctx cancelled:                           report(canceled), exit

    text, calls, err = agent.Send(ctx, directive)   # calls = tool calls this turn
    persist snapshot(transcript, spend, status=running)

    if err is provider rate-limit:  report(usage_limited), exit
    if err (other):                 report(error), exit

    # Cheap spin check, before the verifier: a turn that called no tools did no
    # work, so do not auto-continue on it. Two in a row is a stall. This is the
    # same guard Codex's dispatcher uses (plan-only turns do not continue, and a
    # continuation that makes no tool call suppresses the next one).
    if calls == 0:
        no_op_streak++
        if no_op_streak >= 2:       report(blocked, "no action taken"), exit
        directive = nudge_to_act    # "take a concrete action or report the blocker"
        continue
    no_op_streak = 0

    if blocked(GitSignal(repo)):    report(blocked), exit   # same state 3 turns

    v = verify()                    # command first, then prompt
    if v.passed:                    report(passed, v.by, v.evidence), exit

    directive = audit_directive(v.reason)   # feed the failure back, with the audit

report(budget_limited)   # ran out of iterations without passing
```

`verify()`:
```
if command set:
    run it; if exit != 0: return {passed:false, by:"command", evidence:output}
if prompt set:
    resp = verifier_model(condition=prompt, transcript=agent.Transcript())
    if resp starts FAIL: return {passed:false, by:"prompt", evidence:reason}
return {passed:true, by: prompt? "prompt":"command", evidence}
```

`audit_directive(reason)` is the failure fed back, wrapped in the Codex-style audit:
not done because `<reason>`; address exactly this; show concrete evidence
(command output, exit codes, file contents); do not shrink the objective; do not
take an easier change just to pass; never weaken or delete a test.

## 7. State and resume

After each iteration, write the snapshot atomically to the scoped state path (`.neo/loops/state/<name>.json` for a local loop, `~/.neo/loops/state/<name>.json` for a global one):

```json
{
  "loop": "docs-drift",
  "status": "running",
  "iteration": 2,
  "cost_usd": 1.20,
  "tokens": 28110,
  "transcript": [ ... ],
  "updated_at": "2026-06-22T08:04:11Z"
}
```

`neo loop resume <name>` loads the scoped state file, restores the conversation with the agent's
`ReplaceTranscript`, and continues from the saved budget. This is the durable-state
idea from Codex, without SQLite. A JSON file per loop is enough; do not add a database.

---

## 8. The status state machine

| Status | Meaning | Terminal |
|--------|---------|----------|
| `running` | in progress | no |
| `passed` | the verifier confirmed the condition | yes |
| `nothing_to_do` | the verifier passed before any iteration | yes |
| `blocked` | the same failure 3 turns with no new evidence | yes |
| `budget_limited` | hit max iterations, cost, or time | yes |
| `usage_limited` | the provider rate-limited the account | yes |
| `error` | a non-recoverable failure | yes |
| `canceled` | Ctrl-C or shutdown | yes |

Two specifics worth copying from Codex: keep `budget_limited` (your own cap) and
`usage_limited` (the provider's throttle) as distinct states, by mapping a rate-limit
error from the provider to `usage_limited`. And, if you add `pause`/`resume` later,
protect terminal states so pausing a finished loop cannot un-finish it.

## 9. The run report (JSON)

Printed human-readable, and written as JSON so another process can consume it.

```json
{
  "loop": "docs-drift",
  "status": "passed",
  "verified_by": "command",
  "evidence": "go test ./... ok; check-docs.sh ok",
  "iterations": 2,
  "cost_usd": 1.84,
  "tokens": 41205,
  "elapsed_seconds": 73,
  "artifacts": { "diff": "docs/setup.md | 6 +-", "pr_url": null },
  "blocker": null,
  "next_step": null,
  "started_at": "2026-06-22T08:03:58Z",
  "ended_at": "2026-06-22T08:05:11Z"
}
```

`verified_by` is `command`, `prompt`, or `none` (for non-passed statuses). This is
what makes the loop composable: CI gates on `status == passed`, and a higher loop can
read the report and decide what to do next.

**A non-passing run must hand back what it learned.** As the Codex Goals cookbook puts
it, reaching a budget limit is not the same as completing the objective: a stopped run
should summarize progress, the blocker, and the next useful step. So for any status that
is not `passed`, the runner fills `blocker` (why it stopped, in one line) and `next_step`
(what would unlock progress). This is the difference between a status code and a useful
handoff, and it is exactly what a parent loop or a human needs to act:

```json
{
  "loop": "checkout-latency",
  "status": "budget_limited",
  "verified_by": "none",
  "evidence": "p95 went 180ms -> 134ms over 5 iterations; still above 120ms target",
  "iterations": 5,
  "cost_usd": 4.90,
  "blocker": "remaining latency is in a third-party SDK call, not our code",
  "next_step": "raise max_iterations, or scope the goal to our own code path",
  "started_at": "...", "ended_at": "..."
}
```

## 10. Surfaces: CLI and slash commands

Two front doors to one runner. The CLI is for scripts and CI (prints JSON); the slash commands are for an interactive session (print a live status and a human-readable report).

### CLI

```
neo loop new "<plain description>" [--global]   # draft a loop from a sentence (section 4)
neo loop run <name> [--global] [--max-iterations N] [--max-cost X] [--json]
neo loop list                                   # list defined loops, both scopes
neo loop status <name>                          # show the last run report
neo loop resume <name>                          # continue a crashed/paused run
neo loop schedule <name>                        # install the cron trigger (v2)
```

`neo loop list` shows both scopes, labelled, with last-run status and a shadowing marker so it is clear which file a name resolves to:

```
LOOPS
  local   docs-drift     Fix one doc that drifted from the code     - last: passed (2 iter)
  local   smoke-prod     Curl /readyz and the web URL               - idle
  global  morning-triage Summarize overnight CI failures            - idle
  global  docs-drift     (shadowed by local)
```

For scheduling in v1, do not build a daemon. `neo loop run <name> --json` driven by system cron is the routine. Add `neo loop schedule` later as a convenience.

### Slash commands

```
/loops              list defined, runnable loops with their state
/loop <name>        run a loop by name (local first, then global)
/loop stop          cancel the loop running in this session
```

`/loops` is the picker. It lists what you can run, grouped by scope, with each loop's `description` as the label and its run state inline. This is why `description` is required: name, scope, last-run, and budget can all be read off the file and the state, but the human intent only exists if the author wrote it down.

```
/loops
  LOCAL
    docs-drift      Fix one doc that drifted from the code     - last: passed
    smoke-prod      Curl /readyz and the web URL               - running... 2/5
  GLOBAL
    morning-triage  Summarize overnight CI failures            - idle
```

Note "defined and runnable," not "active." Most loops are not running; they are available. Live state shows inline (`running... 2/5`, `last: passed`). If you ever want only the executing ones, that is a narrower `/loops running` view, but the common need is "what loops do I have."

`/loop <name>` runs the same engine as `neo loop run`, surfaced in the TUI: it shows a live indicator while iterating (a `* <name> active` line, the same pattern Claude Code uses for `/goal`), and Ctrl-C or `/loop stop` cancels it. A bounded loop started in chat is still a multi-iteration job, so the visible stop matters. The slash command prints the human-readable report; the CLI prints the JSON.

One naming note: in Claude Code, `/loop` is the *timer* command. In neo, `/loop <name>` runs a defined loop. That is a deliberate divergence (neo collapses the verbs into one runner), so call it out for anyone coming from Claude Code.

### TUI workflow checklist

Neo already has a visible workflow checklist (`internal/workflow`) that the chat model can update with `create`, `start`, `complete`, `fail`, `skip`, and `clear` events. Use that for interactive loop runs.

The workflow checklist is the live display layer, not the source of truth. The loop runner still owns durable state, budgets, verifier results, reports, and resume data. When a loop runs through `/loop <name>`, a thin adapter publishes loop events into the existing workflow event channel.

```
Loop runner
  -> loop events: preflight, verify, iteration, blocked, report
  -> TUI adapter
  -> workflow.Event
  -> existing visible checklist panel
```

For `/loop docs-drift`, the initial checklist can be:

```
Loop: docs-drift
[pending] Preflight
[pending] Initial verifier
[pending] Iteration 1 / 3
[pending] Verify result
[pending] Final report
```

During the run, item detail carries the useful evidence:

```
[done] Preflight - clean worktree, budget ok
[failed] Initial verifier - check-docs.sh failed
[running] Iteration 1 / 3 - editing docs/setup.md
[running] Verify result - go test ./... passed, prompt verifier running
[done] Final report - passed by command+prompt
```

This makes the loop visibly bounded and inspectable. Budget progress, verification, stalls, and final status all have a place in the UI, and Neo's existing tool/subagent activity can keep attaching to the active workflow item. The CLI does not need this layer; it stays JSON-only.

## 11. How this maps to the code already written

This is an extension of the `internal/loops` package from earlier, not a rewrite.

| Spec piece | Existing code | Change |
|------------|---------------|--------|
| Budget (iterations/cost/time) | `Budget`, `exceeded` | add `max_cost_usd` from a price table |
| Prompt verifier | `LLMEvaluator` + `Transcript` | becomes the `prompt` branch of `verify()` |
| Blocked detection | `GitSignal`, `progressTracker` | emit `blocked` at the 3-turn threshold; add the zero-tool-call check |
| Boundaries enforcement | `permission` policy | deny writes outside `boundaries.write` at the tool layer |
| State + resume | `Snapshot`, `SaveSnapshot` | wire to the scoped state path + `ReplaceTranscript` |
| Run algorithm | `RunGoal` | generalize into `RunLoop(def)` driven by a definition |
| Command verifier | new | shell out, check exit code, capture output |
| Definition loader + validation | new | parse YAML, enforce required fields (section 2) |
| Scope resolver | new | local-then-global lookup, `--global`, scoped state paths |
| `neo loop new` generator | new | inspect repo, draft a file, confirm verifier + budget |
| Slash commands | new | `/loops` picker and `/loop <name>` with a live indicator |
| TUI workflow checklist | `internal/workflow`, TUI workflow panel | publish loop lifecycle events into the existing checklist |
| First-run trust gate | new | confirm a project-local loop's command on first use |
| Skip gate | new | run `verify()` once before the loop |
| Report emitter | new | the JSON in section 9 |

`RunGoal` already is the core run algorithm for the verifier-stop case. The new work is
the definition file and loader, the two scopes, the generator, the command verifier, the
slash commands, the workflow checklist adapter, and the report.

---

## 12. What goal / loop / routine become in neo

There is no separate `goal` or `routine` command. There is one loop, configured:

| The idea | The neo loop definition |
|----------|--------------------------|
| goal (run until verified) | `verify` set, `trigger: manual` |
| timer-loop (watch on a cadence) | `trigger: interval`, stop on budget |
| routine (run while away) | `trigger: schedule` |

If you still want the words for ergonomics, `neo goal "<condition>"` can be sugar that
generates a throwaway definition with a prompt-verifier and a small default budget. But
the trustworthy, teachable unit is the loop file. Lead with that; treat `neo goal` as the
quick, unbounded-feeling shortcut it is.

---

## 13. Build order

- **v1 (the minimum useful shape):** the definition file with validation and the six-part objective, `boundaries` enforced at the tool layer, local + global scopes with local-first resolution, `neo loop new` to generate one from a sentence, `neo loop run`, the `/loops` picker and `/loop <name>`, workflow checklist updates for interactive runs, `trigger: manual`, command and/or prompt verifier (command-first), a budget that refuses empty, the skip gate, blocked via `GitSignal` plus the zero-tool-call check, the JSON report with `blocker`/`next_step`, the per-iteration state snapshot, and the first-run trust gate for project-local loops. Most of this wires the existing `internal/loops` code to a file loader, a scope resolver, a generator, a workflow adapter, and a report emitter.
- **v2:** `neo loop resume`, worktree isolation, `trigger: interval` and `trigger: schedule` with `neo loop schedule`, `pause`/`resume`.
- **Deferred:** parallel loops, a shared artifact/log layer (the outer loop), any database.

Teach the pattern first, in tools people already have. Then present neo's loop runner as one clean implementation of that pattern, with the loop file as the closing demo.
